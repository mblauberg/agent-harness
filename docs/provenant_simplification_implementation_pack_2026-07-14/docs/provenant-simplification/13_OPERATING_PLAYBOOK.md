# Operating playbook

## 1. Advisory request

Example:

> Explain how provider authority currently works.

Chair behaviour:

- read relevant source;
- answer with uncertainty and anchors;
- no heavy delivery run;
- no worker unless repository exploration is large enough to justify one;
- no external effects.

## 2. Routine implementation

Example:

> Correct a local parser bug with existing failing tests.

Flow:

1. Chair confirms objective and current test oracle.
2. Minimal run envelope and exact write scope.
3. One owned workspace.
4. Chair or one implementation worker changes code.
5. Deterministic checks.
6. Review only if policy derives it.
7. Return concise result or PR proposal.

No graph, council or separate session unless needed.

## 3. Substantial implementation

Example:

> Add tenant-scoped rate limiting across API and persistence.

Flow:

1. Readiness or scope.
2. Persist run and authority.
3. Chair decides serial versus independent WorkItems.
4. Allocate exact ownership.
5. Provider-native implementation.
6. Deterministic verification.
7. Derived independent review.
8. Bounded repair.
9. Human acceptance.
10. Typed PR effect.

## 4. Broad scoping

Example:

> Design and implement organisation-level billing.

Flow:

1. Chair explores repository and current decisions.
2. Use bounded research/explorer workers.
3. Present grouped decision packet with recommendations.
4. Record decisions, risks and one-way doors.
5. Produce ScopePacket.
6. Produce Initiative and vertical WorkItems.
7. Human approves scope and authority.
8. Implement ready WorkItems independently where safe.

Do not begin implementation while material owner decisions remain unresolved.

## 5. Crucial change

Example:

> Change the provider authority compiler.

Flow:

- explicit design and alternatives;
- risk floor;
- characterisation tests;
- isolated write profile;
- adversarial containment/security tests;
- other-primary and specialist review;
- bounded repair;
- human acceptance;
- exact PR effect;
- no release/deploy expansion.

## 6. Parallel work

Use parallelism for:

- independent package changes;
- independent research;
- read-only audits;
- separate vertical WorkItems.

Do not parallelise:

- same-file editing;
- tightly coupled debugging;
- decisions that require sequential evidence;
- work with unclear ownership;
- integration without a designated owner.

## 7. Context and handoff

Create durable handoff when:

- chair changes;
- provider changes;
- session context is materially degraded;
- work crosses days;
- the run has multiple owners;
- recovery is required.

Handoff includes:

- objective and non-goals;
- approved and authority digests;
- current state;
- decisions and rejected options;
- artefacts and checks;
- open risks;
- next action;
- remaining budget;
- exact recovery instruction.

## 8. Human communication

Prefer decision packets:

- current conclusion;
- why the human is needed;
- recommended default;
- alternatives and consequences;
- what continues in parallel;
- exact authority or decision requested.

Do not stream raw agent activity as progress.

## 9. Model use

Use the strongest chair where:

- decomposition is difficult;
- ambiguity is high;
- synthesis is broad;
- architecture or weak-oracle judgement matters.

Use smaller models for:

- bounded inventories;
- extraction;
- repetitive transformations;
- independent test-log analysis;
- formatting/schema work;
- narrow review lenses with deterministic return contracts.

## 10. Completion

A run is complete when:

- acceptance criteria map to evidence;
- required checks and review pass;
- no unresolved blocking findings;
- authority and ownership are closed;
- retained artefacts are classified;
- remaining human decision is explicit;
- external effect is either not requested, proposed or exactly executed;
- the status is terminal or awaiting the named human action.
