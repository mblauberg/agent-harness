# Requirements-specification fit

Map diagrams to the target document's own structure. Do not assume fixed
section numbers, release phases, actor tables or marking criteria.

Before authoring, identify:

- the question each diagram must answer;
- the canonical actor, use case and requirement identifiers;
- which textual requirement owns each relationship and flow;
- the project's source, render, caption and index locations;
- the required notation, accessibility and publication profile; and
- whether the request authorises source edits, generated assets and index or
  specification updates.

Use case diagrams normally describe scope and actor goals. Activity diagrams
normally elaborate an important flow, including relevant alternate and failure
paths. The required coverage is a project decision; never infer that a specific
release, number of diagrams or work allocation is mandatory.

Before completion, confirm that actor and use case names agree across the
requirements register, detailed descriptions, diagrams and captions. A render
is supporting evidence, not the source of truth for requirement ownership.
